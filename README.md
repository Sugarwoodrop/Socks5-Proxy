# SOCKS5 Proxy

Реализация SOCKS5-прокси (RFC 1928) на C с использованием `epoll` для мультиплексирования I/O. Однопоточный сервер, обслуживающий произвольное число клиентов через неблокирующие сокеты.

## Возможности

- Метод SOCKS5 `CONNECT` (TCP relay).
- Поддержка адресов: IPv4 (`ATYP=0x01`) и доменное имя (`ATYP=0x03`).
- **Собственный асинхронный DNS-резолвер** поверх UDP — без `getaddrinfo` и сторонних библиотек. Запрос на `8.8.8.8`, разбор ответа с поддержкой DNS-сжатия (compression pointers).
- Корректная работа с неблокирующим `connect`: через `EINPROGRESS` → `EPOLLOUT` → `getsockopt(SO_ERROR)`.
- Backpressure в туннеле: при заполнении буфера снимается `EPOLLIN` с источника, при освобождении — возвращается. Снимает `EPOLLOUT` при пустом буфере.
- Тайм-аут неактивности клиента (60 секунд) с автоматической чисткой.
- Грейсфул-перезапуск по `SIGHUP` (через `execvp`), завершение по `SIGINT`/`SIGTERM`.

## Что не поддерживается

- Аутентификация (метод `0x00 NO AUTH` only) — методы `USERNAME/PASSWORD` и `GSSAPI` не реализованы.
- Команды `BIND` и `UDP ASSOCIATE` — только `CONNECT`.
- IPv6 (`ATYP=0x04`) — возвращает `ATYP not supported`.
- DNS поверх TCP, повторы DNS-запросов, выбор резолвера (захардкожен `8.8.8.8`).

## Архитектура

Однопоточный event loop на `epoll`. У каждого клиента — конечный автомат, разбивающий протокол на состояния:

```
HELLO       (handshake)
   ↓
REQUEST     (парсинг CONNECT-запроса)
   ↓
DNS         (если адрес — домен, асинхронный запрос на 8.8.8.8)
   ↓
CONNECT     (неблокирующий connect к target)
   ↓
REPLY       (отправка SOCKS5-ответа клиенту)
   ↓
TUNNEL      (двунаправленный relay данных с buffering и backpressure)
```

Состояния явно перечислены в `enum`, переходы — внутри функций-обработчиков (`StartUserVerification`, `HelloReply`, `StartSettings`, `SettingsReply`, `TunnelSend`, `TunnelRecv`). Каждый этап читает/пишет ровно столько байт, сколько готов отдать сокет, и сохраняет промежуточное состояние в `ClientSocket` между вызовами `epoll_wait`.

DNS-запрос сериализуется вручную (`CreateDNSpacket` + `encode_dns_name`), отправляется в общий UDP-сокет, который тоже зарегистрирован в `epoll`. Когда приходит ответ — `RecvDNS` ищет клиента по transaction ID и заполняет ему IPv4 в контексте.

## Файлы

| Файл | Назначение |
|---|---|
| `main.c` | Инициализация, главный event loop, диспатч событий по типу fd |
| `ClientStruct.{c,h}` | Структура клиента и его контекстов, lifecycle (init/add/delete/search) |
| `HelloCheck.{c,h}` | SOCKS5 handshake (метод `0x05`, выбор аутентификации) |
| `SocksSettings.{c,h}` | Парсинг CONNECT-запроса и формирование ответа |
| `ConnectServer.{c,h}` | Установление соединения с target, DNS-резолвер |
| `Tunnel.{c,h}` | Двунаправленный relay данных в установленном соединении |
| `SendRecvOneByte.{c,h}` | Хелперы для побайтового чтения/записи с обработкой `EAGAIN` |

## Сборка и запуск

```bash
gcc -Wall -Wextra -O2 *.c -o socks5
./socks5
```

Сервер слушает порт **1080** на всех интерфейсах.

## Тестирование

```bash
# через curl
curl --socks5-hostname localhost:1080 https://example.com

# или настроить системный/браузерный SOCKS5 proxy на 127.0.0.1:1080
```

## Замечания и известные ограничения

- Нет настройки порта/адреса прослушивания через аргументы — захардкожен `0.0.0.0:1080`. Для смены — править `SOCKET_PORT` в `main.c`.
- Размер буфера relay фиксирован в 512 байт. Для трафика с большими пакетами (HTTP/2, видео) стоит увеличить.
- DNS-резолвер не повторяет запросы при потере UDP-пакета — клиент просто получит ошибку connect по таймауту.
- Нет логирования в файл, только `printf`/`perror` в stdout/stderr.

## Что было поправлено относительно изначальной версии

При подготовке этой версии исправлены:

- **Ошибка в стратегии роста буфера клиентов**: условие `sizeClients <= sizeBufClients - 5` срабатывало при пустом буфере, а не при заполненном (направление сравнения было перепутано). Сейчас буфер растёт на +300 элементов при заполнении.
- **Лишние `realloc`** в `addClient`/`DeleteClient`, которые на каждое подключение/отключение перевыделяли буфер ровно под текущий размер — это сводило на нет всю стратегию пред-аллокации. Теперь ёмкостью управляет только `main`.
- **Ручной цикл сдвига буфера** в `TunnelSend` заменён на `memmove` (правильнее по семантике и не копирует мусор за пределами полезных данных).
- **`InitClientSocket`** при ошибке `fcntl` на `socketEndPoint` закрывал не тот файловый дескриптор. Теперь корректно выставляет `socketEndPoint = ERROR`, чтобы вызывающий код мог это обнаружить.
- Унифицированы названия (опечатки в идентификаторах: `Cheak`→`Check`, `Bait`→`Byte`, `Delite`→`Delete`, `Tunel`→`Tunnel`, `Replay`→`Reply`, `Clietn`→`Client`, `Merhods`→`Methods`, `SUCCES`→`SUCCESS`).
- Добавлены include-guards в хедеры.
